============================================================
1. PROJECT SUMMARY
============================================================

Signet is a full-stack signature authentication system that compares two uploaded signature images and returns an authenticity verdict.

Primary objective:
- Verify whether a test signature matches a genuine reference signature.

Core value:
- Provides a practical, explainable signature comparison pipeline with visual evidence (difference heatmap), similarity scoring, and confidence labels.

Technology stack (high level):
- Frontend: Next.js 14, React 18, TypeScript, Tailwind CSS, Framer Motion
- Backend: FastAPI, PyTorch, OpenCV, NumPy
- Deployment: Frontend on Vercel, Backend on Vercel Python deployment


============================================================
2. PROBLEM DOMAIN AND USE CASES
============================================================

Signet addresses digital signature verification workflows where visual comparison and approximate similarity scoring are needed.

Potential use cases:
- Preliminary verification in document workflows
- Educational/demo systems for signature matching concepts
- Decision-support layer before manual human review

Important context:
- Current system output is best treated as decision support, not legal proof.
- Performance quality depends heavily on model weights, data quality, and input consistency.


============================================================
3. SOLUTION OVERVIEW
============================================================

The user uploads:
1) Genuine signature image
2) Test signature image

System workflow:
1. Frontend collects files and submits multipart request.
2. Next.js API route (/api/verify) proxies request to Python backend.
3. Backend preprocesses both images.
4. Backend computes similarity through either neural mode or classical fallback mode.
5. Backend returns:
   - is_authentic
   - similarity_score
   - confidence
   - verdict
   - difference_heatmap (base64 image)
   - inference_mode
6. Frontend displays scores, verdict, and visual comparison.


============================================================
4. ARCHITECTURE DETAILS
============================================================

4.1 Frontend Layer
- Built with App Router architecture in Next.js 14.
- Main UI page in app/page.tsx.
- Uses dedicated reusable components for upload, result card, and similarity meter.
- Calls internal route app/api/verify/route.ts.

4.2 API Proxy Layer
- The Next.js route receives browser requests and forwards them to backend endpoint /verify.
- Backend base URL is controlled with environment variable:
  PYTHON_BACKEND_URL

4.3 Backend Service Layer
- FastAPI service exposes:
  - GET /
  - GET /health
  - POST /verify
- Handles image validation, preprocessing, model inference/scoring, and response formatting.

4.4 ML/Inference Layer
- Siamese-inspired signature comparison pipeline with embedding-based inference when pretrained weights are available.
- Includes classical fallback mode when pretrained weights are unavailable to avoid unreliable random-weight behavior.

4.5 Explainability Layer
- Difference heatmap is generated from pixel-wise differences and returned to frontend for visualization.


============================================================
5. MACHINE LEARNING AND IMAGE PIPELINE
============================================================

5.1 Preprocessing Steps (per image)
- Decode bytes with OpenCV.
- Convert to grayscale.
- Apply Gaussian blur.
- Apply Otsu thresholding.
- Detect signature region via contours.
- Crop and center detected region.
- Resize to standard input size (typically 128x128).
- Normalize values to [0, 1].
- Convert to tensor shape suitable for model inference.

5.2 Siamese Encoder (as documented)
- Convolution + ReLU + MaxPool blocks with increasing channels.
- Adaptive pooling to reduce feature map to compact vector.
- Dense layers to create normalized embedding.
- Shared encoder compares both signatures in same feature space.

5.3 Inference Modes
A) Neural mode
- Used when pretrained weights are successfully loaded.
- Uses embedding similarity.
- Typical thresholds (documented):
  - >= 0.88: Genuine
  - 0.78 to 0.8799: Suspicious
  - < 0.78: Forged

B) Classical fallback mode
- Used when pretrained weights are not available.
- Uses blended structural and pixel similarity:
  score = 0.7 * SSIM + 0.3 * pixel_agreement
- Stricter thresholds (documented):
  - >= 0.93: Genuine
  - 0.82 to 0.9299: Suspicious
  - < 0.82: Forged

5.4 Output Semantics
- similarity_score: Numeric similarity estimate.
- confidence: Qualitative confidence level.
- verdict: Human-readable class label.
- difference_heatmap: Visual guidance for mismatched regions.
- inference_mode: Clarifies whether neural or fallback path was used.

============================================================

- verdict (string)
- difference_heatmap (base64 data URL)

Error scenarios:

- Unsupported format
- Preprocessing failure
- Inference runtime error
